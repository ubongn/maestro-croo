## Dual-Role Event Routing

Maestro operates as **both provider and consumer** on a single WebSocket connection. This creates a routing challenge: `OrderPaid` fires when a user pays Maestro AND when Maestro pays a sub-agent.

### Solution: Shared Outbound Set + State Tracking

```
Event: negotiation_created
  в”Ӯ
  в”ңв”Җ Is negotiationId in outboundNegotiations?
  в”Ӯ   в”ңв”Җ YES вҶ’ Consumer originated this. Remove from set. Skip.
  в”Ӯ   в””в”Җ NO  вҶ’ Incoming user order. Provider accepts it.
  в”Ӯ
Event: order_paid
  в”Ӯ
  в”ңв”Җ Is orderId in provider.providingOrders?
  в”Ӯ   в”ңв”Җ YES вҶ’ User paid Maestro. Trigger orchestration.
  в”Ӯ   в””в”Җ NO  вҶ’ Maestro paid a sub-agent. Consumer collects delivery.
```

## Race-Safe Event Bus

The `CrooBus` uses a short TTL buffer + `waitFor(predicate)` pattern to handle
the gap between `negotiateOrder()` returning and a listener registering:

```

    в”Ӯ
    в”ңв”Җв”Җв–ә order_created event arrives (before listener is ready)
    в”Ӯ         в”Ӯ
    в”Ӯ    в”Ңв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯ    в”Ӯ  Buffer   в”Ӯ  (TTL: 30s)
    в”Ӯ    в”Ӯ  [event]  в”Ӯ
    в”Ӯ    в””в”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”ҳ
    в”Ӯ         в”Ӯ
    в”ңв”Җв”Җв–ә waitFor(orderId matching negotiationId) вҖ” finds buffered event
    в”Ӯ
    в””в”Җв”Җв–ә proceed to payOrder()
```
